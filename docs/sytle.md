目标

本文件定义 yytools 的命名、API 设计、错误/断言策略、注释模板与代码组织规范。目标是降低长期维护成本，提高搜索/补全友好性，并确保行为可预测。

命名规范

包命名
•	使用短小、可读、语义明确的名词
•	避免 util / helper 这种“语义垃圾桶”命名
•	推荐使用扩展库风格：mathx、slicex、stringx

文件命名
•	按领域拆分（便于定位与演进）
•	示例：gcd.go、minmax.go、slice_select.go
•	注：现有 mathx/common.go 是历史积累；新增代码或重构时应按领域拆分。

函数命名

1）语义前缀聚合（搜索/补全友好）
•	同一语义族使用统一前缀：Min*、Max*、Gcd*、Abs*
•	对称函数成对出现：Min/Max、MinInSlice/MaxInSlice、MinBy/MaxBy

2）后缀表达关键语义（防歧义）
•	*InSlice：说明作用对象是 slice
•	*By：说明通过比较器/评分函数决定“更优”
•	*OK：说明返回 (…, ok bool)，避免断言/panic
•	*I/*R：迭代/递归实现（如 GcdI/GcdR）

API 设计规范

Assert 版与 OK 版双轨策略

Assert 版（强约束）
•	用于算法/数据结构内部或明确不应出现空输入/非法参数的场景
•	行为：非法输入触发 assert.Assert(...)

示例：
•	MinInSlice(s []T) (idx int, v T)：空 slice 断言

OK 版（容错）
•	用于 gameplay/业务层，输入可能为空或条件不足需要分支处理
•	行为：返回 ok=false，不触发断言

示例：
•	MinInSliceOK(s []T) (idx int, v T, ok bool)

规则：OK 版必须由内核实现；Assert 版必须是 OK 版的薄封装。
目的：避免重复逻辑导致行为漂移。

稳定性（Stability）要求

当存在“并列最优”时（例如多个元素同样最小），默认要求返回 第一个最优元素，并在注释中明确：
•	“保证稳定：返回第一个并列最优元素”

如果函数不保证稳定，必须明确说明并给出原因。

比较器/评分函数约束

对 *By 系列：
•	better(a, b) == true 表示 a 优于 b，应替换当前最优值
•	建议 better 满足传递性；若不满足，结果仍定义为“按扫描顺序得到的局部最优”，但应在注释中提示使用者注意

对 Ordered 系列：
•	若 T 可能包含 float32/float64，则应在文档中提示 NaN 非全序行为（与 Go < > 一致）

错误处理规范
•	algorithms/ds 内部倾向使用 assert（快速暴露 bug）
•	gameplay（L4）/ 对外 API 倾向返回 ok 或 error，由调用方决定策略
•	不在底层库内部直接 panic（除非是不可恢复的程序错误且已明确写入文档）

注释模板（推荐）

纯函数/算法
•	一句话描述
•	输入约束（是否允许空/负数/零）
•	返回值含义
•	稳定性承诺（如适用）
•	复杂度（推荐）

示例模板：
•	功能：……
•	输入：……
•	返回：……
•	稳定性：……
•	复杂度：时间 O(…)；空间 O(…)

组件/结构体
•	组件职责
•	并发语义（线程安全？调用约束？）
•	生命周期（Start/Stop/Close）
•	资源占用（goroutine、timer、内存）
•	可观测钩子（logger/metrics）

代码组织与演进

内核函数收敛原则
•	所有逻辑应尽量收敛到一个或少数“内核实现”
•	导出函数优先作为薄封装（参数检查、语义分层、返回格式）

兼容性原则
•	已导出的函数签名尽量不破坏
•	新需求优先新增函数（例如增加 OK 版本）
•	需要弃用时，用 Deprecated: 注释，保留 wrapper，并给出替代函数

测试与基准（推荐）
•	algorithms：单元测试覆盖边界（空、极小、极大、并列值）
•	ds/infra：提供 benchmark（至少一个基准入口）
•	gameplay（L4，规划中）：用可注入依赖保证可测（配置驱动、可注入时钟/随机源/外部接口）
