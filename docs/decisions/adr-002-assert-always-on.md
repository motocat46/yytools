# ADR-002：断言框架始终启用，不提供运行时开关

**状态**：已采纳
**日期**：2026-03
**影响范围**：`pkg/common/assert/`，所有使用 `assert.Assert` 的包

---

## 背景

`assert` 包经历了三个版本（详见 `pkg/common/assert/DESIGN.md`）。V2 引入了 build tag 控制开关，但代码审查中反复出现如下循环：

1. 发现 `assert.Assert(...)` 守护安全关键检查
2. 判断"如果断言被关闭，此处会静默失效"
3. 将其改为无条件 `panic`

这个循环在 Stack、Queue、GcdR、RadixSort 等多处重复，是明显的设计信号。

---

## 决策

**断言框架始终启用，删除所有运行时/编译时开关。**

`assert.Assert(cond, msg)` 是 `if !cond { panic(msg) }` 的正向条件语法糖，没有任何开关语义。

---

## 理由

### 1. 性能理由不成立

断言守护的条件（`length > 0`、`n >= 1`、`ptr != nil`）全是 O(1) 比较，分支预测后开销可忽略不计。基准测试显示断言路径无额外内存分配。

### 2. 可关闭的前提自我矛盾

"生产环境关闭断言"的隐含前提是"代码已充分测试"——但如果代码正确，断言永远不触发，关闭没有任何意义；如果代码有 bug，关闭断言让 bug 在生产静默传播，代价远高于偶发的 panic。

### 3. 消除"用哪个"的决策成本

V2 引入了两个路径：用 `assert.Assert`（可能被关闭）还是用无条件 `panic`（永远有效）。每次写代码都要做这个选择，且审查者需要反向验证选择是否正确。V3 消除这一分歧：所有调用方 bug 用 `assert.Assert`，运行时边界用无条件 `panic`，外部输入用 `error`。

---

## Assert vs panic 使用边界（V3 确立）

| 场景 | 使用 |
|------|------|
| 调用方违反契约（空栈出栈、负数索引等） | `assert.Assert` |
| 运行时边界保护（时间戳溢出、类型范围超限） | 无条件 `panic` |
| 外部输入校验 | 返回 `error` |
