# sort 设计记录

面向维护者，记录关键决策及其背后的理由。使用文档见 README.md。

---

## 一、算法组合的设计思路

本包收录了多种排序算法，不是"大而全"，而是每种算法在特定场景下有独特价值：

| 算法 | 独特价值 |
|------|---------|
| `QuickSort` | 通用首选；三路划分优化重复元素；随机化 pivot 防止有序输入退化 |
| `QuickSortTraversal` | 迭代版快排，消除递归调用栈，适合超大数组（防栈溢出） |
| `CountingSort` | 值域集中时 O(n+k) 远快于 O(n log n)；整数排名/计分类场景的专用工具 |
| `RadixSort` | 多位整数的稳定线性排序 |
| `InsertionSort` | 小数组（≤12）的实际最优；快排内部自动切换到它 |
| `SelectionSort` | 教学目的，无实际性能优势 |

## 二、快排的关键优化决策

### 决策一：三路划分（荷兰国旗算法）替代二路划分

经典二路划分在大量重复元素时退化为 O(n²)（所有等于 pivot 的元素都落入同一侧）。

三路划分将数组分为三段 `< pivot | == pivot | > pivot`，等于 pivot 的部分直接跳过递归，重复元素越多提升越明显：全相同元素时从 O(n²) 降到 O(n)。

### 决策二：随机化 pivot

`partition3way` 每次随机选取 pivot 而非固定取首/尾元素，将有序/逆序输入退化 O(n²) 的概率降到可忽略不计。

### 决策三：小数组切换插入排序（阈值 12）

快排在小数组上的递归开销相对于比较次数太高；插入排序在小数组上常数项更小，且对近乎有序的小数组接近 O(n)。

阈值 12（`maxInsertion = 12`）是工程经验值，Go 标准库 `pdqsort` 使用类似策略。

### 决策四：提供迭代版（Traversal 变体）

递归快排在极端退化输入下可能导致调用栈溢出（Go 默认 goroutine 栈可增长，但仍有上限）。迭代版用显式 `Stack` 模拟递归帧，消除这一风险，适合不确定输入规模的生产场景。

## 三、CountingSort 的边界保护

CountingSort 的额外内存为 `O(max - min + 1)`，差值过大会导致 OOM。

```go
const maxRange = 1e7
if uint64(max-min) > maxRange {
    panic("CountingSort: max-min 差值超出限制，请改用 QuickSort")
}
```

**为什么用 panic 而不是 error？**

排序是纯计算函数，返回 error 会破坏调用方的代码风格（所有排序都要处理 error），而差值过大是**调用方 bug**（选错了算法），用 panic 能在开发阶段立刻暴露，比 error 更合适。

## 四、类型约束：只支持 `base.Integer`，不支持浮点

所有排序函数的类型约束是 `base.Integer`（整数），不包含 `float32`/`float64`。

**原因**：`CountingSort` 和 `RadixSort` 的实现依赖整数的位模式和范围计算，对浮点数没有意义。统一约束到 `Integer` 避免误用——浮点排序直接用标准库 `slices.Sort` 即可（README 中已注明）。

---

## 五、RadixSort 的负数限制

`RadixSort` 当前仅支持非负整数，传入负数会 panic。

**原因**：基数排序基于数字的位模式（从低位到高位逐步排序），负数的补码表示会打乱高位的顺序关系，需要额外处理符号位。当前实现未处理，以 panic 明确限制，而不是悄悄产生错误结果。

未来如需支持负数，可采用"偏移量"技巧（将所有元素偏移为非负数后排序，再还原）。
