## 一、Fibonacci 的双实现：查找表（FibN）与备忘录（Fibonacci.Calculate）

### 两种实现并存的原因

| 维度 | `FibN[T](n)` 查找表 | `Fibonacci[T].Calculate(n)` 备忘录 |
|------|---------------------|--------------------------------------|
| 时间复杂度 | O(1) | O(n)（首次），O(1)（缓存命中）|
| 空间复杂度 | 编译期静态，零运行时分配 | 运行时动态，O(n) heap |
| 适用场景 | 高频调用、已知类型、n 在类型范围内 | n 不确定或需动态计算 |
| 类型安全 | 编译期验证数组值不溢出各自类型 | 运行时 assert 检查 |
| 最大 n | 受类型 T 的 Max 限制（如 int8 最多 n=11）| 同上，由 `FibNMax[T]()` 查询 |

两种实现面向不同调用方：能确定类型和范围时用 `FibN`（零分配、O(1)）；需要动态积累或类型无法预知时用 `Fibonacci.Calculate`（动态 memo）。

### 为什么查找表用 8 个类型化数组而非一个 uint64 数组

**单 uint64 数组**的代价：越界值（如 144 对 int8）在数组中合法存在，编译器无法发现；错误降级为运行时 panic。详见 `fib_table.go` 开头注释。

**8 个类型化数组**的优势：编译器直接验证每个值不溢出各自类型（如 `[12]int8{..., 144}` 编译即报错）。内存差异（~2 KB vs ~752 B）可忽略，但静态安全性更高。

## 二、Abs 的最小有符号整数行为

`Abs(MinInt8)` → 仍返回 `MinInt8`（-128），因为 `-(-128)` 在 int8 补码中回绕为 -128 本身。

这是刻意保持与 Go 内置运算一致的行为，而非 bug。调用方若需处理此情形，应升级类型精度后再取绝对值。

## 三、GCD 的递归与迭代双实现

`GcdR`（递归）和 `GcdI`（迭代）均暴露，供需要递归版本的调用方使用。`Gcd` 默认委托给迭代实现（O(1) 栈空间），与递归版本结果完全相同。

## 四、PowMod / Comb / CombTable 设计决策

### 为何组合数逆元选择 Fermat 小定理

当前组合数接口面向常见竞赛/工程质数模（如 `1_000_000_007`、`998_244_353`）。
在此前提下，逆元可由 `x^(p-2) mod p` 直接得到，实现只依赖 `PowMod`，代码短、边界清晰、易审计。
扩展 GCD 虽可支持更广泛模数，但会引入额外分支与更复杂的正确性面。

### 为何显式限制 `n < mod` / `maxN < mod`

阶乘取模方案在 `n >= mod` 时会遇到 `fact[n] % mod == 0` 的问题，此时直接套用 `n! / (k!(n-k)!)` 与 Fermat 逆元会产生静默错误。
与其返回“看似合法、实际错误”的结果，当前实现直接把正确性边界收窄到 `n < mod` 和 `maxN < mod`。
若业务需要处理更大 `n`，应改用 Lucas 定理，而不是放宽当前实现约束。

### CombTable 为何只做一次 PowMod

朴素做法可对每个 `fact[i]` 分别调用 `PowMod(fact[i], mod-2, mod)` 求逆元，总复杂度约 `O(n log mod)`。
当前实现只对 `fact[maxN]` 调用一次 `PowMod`，再利用：

`invFact[i] = invFact[i+1] * (i+1) % mod`

逆向推导全部逆阶乘，总复杂度降为 `O(n + log mod)`。

### 为何热路径使用 AssertFast

`PowMod`、`Comb` 与 `CombTable.C` 都是高频数值查询函数。
若使用带变参消息的 `assert.Assert`，基准会出现额外分配；改用 `assert.AssertFast` 后，查询路径可稳定做到 `0 allocs/op`。

### int64 乘法的溢出边界

`PowMod`、`Comb`、`CombTable` 都依赖 `int64` 直接做模乘。
为保证中间乘积不溢出，需满足 `mod^2 < 2^63`，即 `mod` 量级应小于约 `3e9`。
常用质数模 `1_000_000_007` 与 `998_244_353` 都在安全范围内。
