# unionfind 设计文档

## 设计目标

提供泛型并查集（Disjoint Set Union），支持任意 `comparable` 类型元素，O(α) 均摊的 Union/Find/Connected/Size/Count，元素首次使用时自动注册。

---

## 核心结构选型

### 为什么用双 map 而不是节点指针？

| 方案 | 实现 | 特点 |
|------|------|------|
| 双 map | `parent map[T]T` + `size map[T]int` | 泛型友好，无额外类型约束，分配集中在注册时 |
| 节点指针 | `*node[T]` 含 parent 指针和 size 字段 | 需要 comparable + 节点分配，查找需要两次间接寻址 |

对于泛型 `comparable` 类型，节点指针需要额外的内存分配（每注册一个元素分配一个 node），且每次 Find 需要两次指针解引用。双 map 方案将所有数据紧凑存放在 map 中，稳态操作（路径已压缩后）只有 map 查找，0 allocs。

### 为什么 Find 用迭代两遍而不是递归一遍？

递归路径压缩实现更简洁：

```go
// 递归版（简洁但有栈溢出风险）
func find(x T) T {
    if parent[x] != x {
        parent[x] = find(parent[x])
    }
    return parent[x]
}
```

但对于极深的初始树（N=10^6 且所有节点链式连接），递归深度可达 N，触发 goroutine 栈溢出。迭代两遍方案无此风险：第一遍找根，第二遍压缩路径，等价语义、零额外分配。

### 为什么选按大小合并而不是按秩合并？

两者均摊复杂度相同（O(α)），但按大小合并：
- Size 字段天然是有用的业务信息（组的实际成员数），无需额外维护
- 按秩合并需要一个只用于控制合并方向的秩字段，对调用方无意义

---

## 关键设计决策

### 自动注册（auto-register）

所有公开方法在访问元素前调用 `register(x)`：未见过的元素自动成为单独组，Count 递增 1，调用方无需显式 Add。

优点：调用方代码更简洁，无"忘记 Add 导致 panic"的陷阱。
代价：每个新元素首次使用时有 2 次 map 写入（parent + size）。

### 不实现 Members(a T) []T

枚举组成员需要 O(N) 全表扫描，语义上是反模式（并查集的优势在于不关心具体成员）。调用方如需枚举成员，自行维护 `map[T][]T` 侧表。

### 不实现 Remove(a T)

并查集经典不支持删除。支持删除需要额外的 virtual node 或 offline 技术，复杂度大幅上升，且使用场景极少。

---

## 复杂度

| 操作 | 时间（均摊） | 空间 |
|------|------------|------|
| Union | O(α(n)) | O(1) |
| Find | O(α(n)) | O(1) |
| Connected | O(α(n)) | O(1) |
| Count | O(1) | O(1) |
| Size | O(α(n)) | O(1) |
| 整体空间 | — | O(n) |

α(n) 为反阿克曼函数，实际使用中 ≤ 4，视为常数。
