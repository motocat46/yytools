## 一、基于 slice 而非链表

栈的两种经典实现：

| 实现 | Push/Pop | 内存布局 | 缓存友好 |
|------|---------|---------|---------|
| slice（当前）| 均摊 O(1) | 连续 | ✓ |
| 链表 | O(1) | 分散 | ✗ |

slice 的连续内存对 CPU 缓存更友好，且不需要每个元素额外分配一个节点对象。Go 的 `append` 自带扩容语义，Push 实现极简。

---

## 二、缩容策略：元素数 < 容量 1/4 时缩为一半

`Pop` 后若 `len < cap/4`，则将容量缩为 `cap/2`，下限 `DEFAULT_STACK_SIZE = 16`。

**为什么是 1/4 而不是 1/2？**

若在 `len == cap/2` 时立即缩容，缩容后容量恰好等于元素数量，紧接着一次 Push 就会触发扩容，产生"扩容 → 缩容 → 扩容"的抖动（hysteresis 问题）。

选 1/4 作为触发阈值，缩容后 `len ≈ cap/2`，留有足够缓冲，避免在边界附近频繁扩缩。这与 `pkg/ds/queue` 采用相同策略保持一致。

**缩容正确性断言**

```go
n := copy(newItems, this.Items)
assert.Assert(n == len(this.Items), "缩容不能改变元素数量!")
```

`copy` 的实际复制数量取决于两个切片中较短的那个，若目标切片容量计算有误，`n` 会小于期望值，元素静默丢失。断言让这类 bug 在开发阶段立刻暴露。

## 三、空栈 API 行为：panic，不返回零值

`Pop()` 和 `Top()` 返回泛型 `T`，零值可能是合法元素，原因与 `queue` 相同（详见 ADR-001）。

```go
// ✅ 正确
if !stack.Empty() {
    item := stack.Pop()
}
```

## 四、弹出后清零槽位

```go
var defaultVal T
this.Items[length-1] = defaultVal
```

弹出后将末尾槽位置为零值，目的是让 GC 能回收该槽位指向的对象（若 T 是指针或含指针的结构体）。不清零的话，底层数组仍持有引用，即使逻辑上已弹出也无法被回收。这是 Go slice 操作的常见陷阱。
