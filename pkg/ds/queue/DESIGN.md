## 一、为什么用环形缓冲区而不是 slice append

最直观的队列实现是 `[]T` + append，但有一个问题：每次 `Dequeue` 后 `head` 移动，切片左侧的空间永久废弃——随着持续的入队/出队，底层数组会不断向右延伸，无法复用左侧空间。

环形缓冲区（Ring Buffer）用取模运算让 `Head` 和 `Tail` "绕圈"，左侧出队留下的空位会被后续入队复用，实现真正的 O(1) 入队/出队，且内存复用率高。

**代价**：需要区分"队列满"和"队列空"（两者在朴素实现中都表现为 `Head == Tail`）。解决方案是**留一个空位**：`Tail` 始终指向下一个可入队的空位，`(Tail+1) % Capacity == Head` 表示满。这使得实际可用容量是 `Capacity - 1`，但消除了状态歧义。

## 二、扩缩容策略

### 扩容：满时翻倍

触发条件：`(Tail+1) % Capacity == Head`（队列满）。
翻倍是常见策略，均摊 O(1) 复杂度。

### 缩容：元素数 < 容量 1/4 时缩为一半

**为什么是 1/4 而不是 1/2？**

若在元素数降到 1/2 时立即缩容，刚缩容完紧接着一次入队就会触发扩容，出现扩容-缩容-扩容的抖动（hysteresis 问题）。

选择 1/4 作为缩容阈值，缩容后元素数约占新容量的 1/2，留有足够缓冲，避免频繁切换。这是教科书推荐的策略。

### 最小容量：16

缩容下界固定为 16，防止小队列频繁在极小容量间扩缩。

## 三、空队列 API 行为：panic，不返回零值

`Dequeue()` 和 `Peek()` 返回泛型 `T`，零值（如 `0`、`""`、`false`）可能是合法元素，调用方无法区分"真实零值"和"空队列信号"——返回零值是静默错误，比 panic 更危险。

```go
// ✅ 正确用法
if !q.Empty() {
    item := q.Dequeue()
}

// ❌ 危险：不检查直接取，0 可能是真实元素
item := q.Dequeue()
```

panic 迫使调用方写 `Empty()` 检查，让 bug 在开发阶段立刻暴露。

## 四、内部不变量

`Empty()` 中通过 `assert` 验证 `Length` 字段与 `Head`/`Tail` 的几何计算结果一致：

```go
assert.Assert(this.calcLen() == this.Length, "队列内部状态不一致")
```

这是一个防御性不变量检查：`Length` 和 `calcLen()` 是两条独立的计数路径，若出现 bug 导致两者分叉，断言会立即暴露，而不是让错误悄悄传播。

`resize` 后也有断言 `!this.full()`，确保扩缩容逻辑正确。

## 五、环绕边界的处理

当队列中存在环绕（`Tail < Head`）时，元素在底层数组中被分为两段：

```
[--后半段--|...空...|--前半段--]
 0        Tail    Head     Cap
```

`copyTo`（扩缩容时调用）、`Range`（遍历）都需要处理这种情况，分别拷贝/遍历两段。注释中有详细说明，修改时需同步更新两处逻辑。
